Motion

From prompt to production — then to photoreal.

Motion is a live prompt-to-video system with a growing knowledge layer: exhaustive named registries that teach a precise interpreter what “ocean,” “jumping fish,” and “golden hour” should mean. We already ship controllable mini-scenes. The next phase adds photoreal generation on top of that same interpreter — not a black-box toy, a production stack.

02

The problem

Text-to-video is exploding — but most tools are either pretty and uncontrollable, or controllable and not photoreal. Creators and teams still can’t say what they mean and get a reliable production.

  • Vague prompting. “Cinematic ocean scene” means something different every run. No shared vocabulary between user language and the engine.
  • Black-box generation. Diffusion APIs look great until you need the same character, setting, motion, and beat structure on take two.
  • No learning loop. Products generate clips; they don’t systematically grow a registry of named visual, audio, and narrative primitives that improve the next prompt.

03

The solution

Motion turns a prompt into a structured scene — then renders it. Today that render is procedural and stylized. Tomorrow it can also be photoreal. The interpreter stays the same.

  1. Interpret Prompt → entities, setting, motion, music, gags, beat windows
  2. Ground Match against Pure / Blended / Semantic / Interpretation registries
  3. Render Procedural engine now; photoreal diffusion as a second engine next
  4. Learn Human thumbs + automatic discovery grow the registries every run

Live product: Cloudflare Worker + D1/R2/KV, automated learning loops, mini-scenes gallery, and a Video AI lab — already in production at motion.productions.

04

Why this is hard to copy

The moat is not “we wrap an API.” It’s a compounding knowledge graph of named video primitives — color, sound, motion, settings, props, expression — wired into creation and review.

Registries as product

Exhaustive named coverage across Pure, Blended, Semantic, and Interpretation — the substrate for a precise prompt interpreter.

Closed learning loop

Automated discovery plus human thumbs-up/down teach the loop what to exploit, demote, and mutate next.

Scene graph, not soup

Settings spawn trees, fish, waves, buildings; characters have expression and personality; beats have time windows.

Engine-agnostic plan

The same structured plan can drive procedural frames today and a photoreal diffusion backend tomorrow.

05

What’s already live

This is not a slideware demo. The stack ships video, learns from it, and improves the next prompt.

  • Production site and APIs on Cloudflare (Worker, D1, R2, KV)

  • Entity / gag / expression discovery into learned_entities
  • Public review gallery with feedback that teaches exploit vs demotion
  • Balanced automation loop continuously exploring and exploiting

Proof surfaces: /mini-scenes/ · /registries/ · /video-ai/

06

The photoreal phase

Procedural Motion stays the controllable, cheap, always-on layer. Photoreal is a second engine — fed by the same interpreter and registries.

Phase 1 · Hybrid API

Structured scene plans compile into diffusion prompts. Ship photoreal clips via a hosted video model while we validate quality and unit economics.

Phase 2 · Owned inference

Local / owned GPU for inference and light fine-tunes (LoRA). Lower marginal cost; brand-consistent look.

Phase 3 · Fine-tune

Adapters on open video models taught by our registries and liked mini-scenes — Motion’s aesthetic, not a generic model dump.

We are not training a foundation model from scratch. We are buying the right compute and API runway to put photoreal behind an interpreter we already own.
